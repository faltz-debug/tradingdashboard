1. PROBLEMAS CRITICOS (Corrigir Imediatamente)

CRITICO C1 — Credenciais expostas no repositorio

O arquivo .env esta commitado no Git com chaves reais: TWELVE_DATA_KEY, TELEGRAM_TOKEN, TELEGRAM_CHAT_ID.

Risco: Qualquer pessoa com acesso ao repositorio pode controlar seu bot Telegram e consumir seus creditos da Twelve Data.

Acao: (1) Revogar e regenerar todos os tokens. (2) Remover .env do historico Git com git filter-repo. (3) Confirmar que .env esta no .gitignore (esta, mas o arquivo ja foi commitado antes).

CRITICO C2 — Calculo de Divergencia invertido

Arquivo: server.js linhas 486-542

A deteccao de divergencia confunde bearish com bullish hidden. Quando o preco faz Higher High e o MACD faz Lower High, o codigo rotula como bearish_macd, mas isso e na verdade uma divergencia bullish hidden (continuacao de alta).

Impacto: Traders recebem sinais de direcao oposta ao que o padrao indica, podendo entrar trades contra a tendencia.

CRITICO C3 — Marcadores SL/TP invertidos no Backtester (SHORT)

Arquivo: backtester.html linha 1483

Para trades SHORT, o marcador de saida aparece belowBar quando deveria ser aboveBar. Alem disso, o backtester nao renderiza linhas horizontais de SL/TP no grafico — so mostra na tabela.

Impacto: O que o usuario ve no grafico nao bate com os numeros da tabela, como voce mesmo observou.

CRITICO C4 — Race condition no estado de alertas

Arquivo: server.js linhas 819-831

O estado dos alertas e atualizado em memoria e salvo no disco com debounce de 500ms. Se o servidor crashar nesse intervalo, alertas duplicados ou perdidos ocorrem no restart. Multiplas escritas concorrentes podem corromper o JSON.

2. ALTA PRIORIDADE

ALTO H1 — MACD Signal Line sem inicializacao correta

A signal line comeca do primeiro valor do MACD array em vez de usar SMA(9) dos primeiros 9 valores como inicializacao. Produz valores diferentes do TradingView/MT5 nos primeiros ~50 candles.

ALTO H2 — Sem retry no loop principal de alertas

Se OANDA/Yahoo/Kraken falhar, o erro e logado mas nao ha retry, backoff exponencial ou alerta ao trader. O sistema degrada silenciosamente.

ALTO H3 — Validacao de dados Kraken/Yahoo ausente

Arrays recebidos das APIs nao sao validados (NaN, null, formato errado). Um candle corrompido propaga para todos os indicadores.

ALTO H4 — VIP: sem validacao de formulario

parseFloat() em campos vazios retorna NaN. Sem verificacao de SL/TP no lado correto da entrada (BUY: SL < Entry < TP).

ALTO H5 — Telegram falha silenciosamente

Muitos chamadores usam .catch(() => {}) — se o token expirar ou o chat ID mudar, alertas param sem aviso.

ALTO H6 — VIP: timeframe bias nao validado

Se chamar /api/vip/signal?entryTf=1h&biasTf=15m, a logica inverte (bias menor que entry), produzindo sinais errados.

ALTO H7 — CORS wildcard em producao

FRONTEND_ORIGIN=* permite requisicoes de qualquer dominio. Em producao, deve ser restrito ao dominio do Railway.

4. O QUE ESTA BEM FEITO

BOM Cadeia de fallback robusta

getAsset() implementa: Cache RT → Cache normal → Mercado fechado → OANDA → Yahoo → Twelve Data → Simulacao. O sistema sempre tem dados, mesmo com multiplas falhas de API.

BOM Confluencia multi-timeframe

Valida que 15M + 1H + 4H concordam antes de elevar a confianca do sinal. Reduz falsos positivos significativamente.

BOM Filtro ADX para mercados laterais

ADX < 25 suprime alertas de tendencia. Evita operar em mercados sem direcao (chop). Pratica profissional.

BOM Calendario economico integrado

Buffer de 30 minutos antes/depois de noticias de alto impacto (NFP, CPI, FOMC). Excelente gestao de risco.

BOM Rate limiting nativo sem dependencias

Implementacao limpa com cleanup automatico de IPs antigos e headers Retry-After corretos.

BOM Cooldown de alertas Telegram

1h entre alertas do mesmo ativo previne spam. Estado persistido no disco sobrevive restarts.
